# PRD - TastePick

## 1. 문서 목적
이 문서는 Android 기반의 간단한 음식 추천 앱 `TastePick`의 MVP 요구사항을 정의한다.
본 문서는 기획 의도와 사용자 시나리오를 명확히 하여 후속 TRD, design-intent, code-quality-guide 생성의 입력으로 사용한다.

## 2. 제품 개요
- 제품명: `TastePick`
- 한 줄 설명: 사용자의 현재 기분, 식사 상황, 알레르기 정보를 바탕으로 오늘 먹을 메뉴를 추천하는 Android 앱
- 대상 사용자:
  - 점심/저녁 메뉴 선택이 자주 어려운 직장인
  - 빠르게 후보 3개 정도만 받고 싶은 사용자
  - 취향과 제외 재료를 앱에 저장해 반복 입력을 줄이고 싶은 사용자

## 3. 문제 정의
- 사용자는 "오늘 뭐 먹지?"를 반복적으로 고민한다.
- 지도/배달앱은 정보가 너무 많아 빠른 의사결정에 오히려 시간이 걸린다.
- 매번 선호도와 제외 재료를 다시 입력하는 과정이 번거롭다.

## 4. 목표
### 4.1 MVP 목표
- 30초 안에 추천 결과 3개를 보여준다.
- 선호도와 제외 재료를 저장해 다음 추천에 반영한다.
- 오프라인 상태에서도 로컬 데이터셋 기준 추천이 가능해야 한다.

### 4.2 비목표
- 결제 기능
- 배달 주문 연동
- 실시간 위치 기반 상점 검색
- 회원가입/클라우드 동기화

## 5. 핵심 사용자 시나리오
### 시나리오 A - 첫 실행
1. 사용자가 앱을 처음 실행한다.
2. 앱은 선호 음식 카테고리, 맵기 허용 범위, 제외 재료, 알레르기 정보를 입력받는다.
3. 입력 완료 후 홈 화면으로 이동한다.

### 시나리오 B - 빠른 추천
1. 사용자가 홈 화면에서 현재 식사 상황을 선택한다.
   - 식사 시간: 아침 / 점심 / 저녁 / 야식
   - 기분: 든든한 / 가벼운 / 매운 / 국물 / 랜덤
   - 예산: 저가 / 보통
2. 사용자가 `추천 받기`를 누른다.
3. 앱은 로컬 데이터셋에서 조건에 맞는 후보를 점수화해 상위 3개를 표시한다.

### 시나리오 C - 피드백 반영
1. 사용자가 추천 결과에서 `좋아요` 또는 `별로예요`를 선택한다.
2. 앱은 해당 피드백을 다음 추천 점수 계산에 반영한다.

### 시나리오 D - 제외 재료 보호
1. 사용자가 땅콩 알레르기 또는 제외 재료를 등록한다.
2. 추천 결과에는 해당 재료가 포함된 메뉴가 노출되지 않는다.

## 6. 기능 요구사항
### 6.1 필수 기능
- 온보딩 선호도 설정
- 홈 화면 추천 조건 입력
- 추천 결과 3개 표시
- 추천 후보 상세 정보 표시
  - 메뉴명
  - 카테고리
  - 예상 가격대
  - 주요 재료
  - 추천 이유
- 사용자 피드백 저장
- 설정 화면에서 선호도 수정

### 6.2 데이터 요구사항
- 앱은 최초 실행 시 내장 JSON 데이터셋을 로컬 DB로 시드한다.
- 메뉴 항목은 다음 속성을 가져야 한다.
  - `id`
  - `name`
  - `category`
  - `mealTimeTags`
  - `moodTags`
  - `priceTier`
  - `ingredients`
  - `spicyLevel`

## 7. 화면 요구사항
### 7.1 온보딩
- 선호 음식 카테고리 복수 선택
- 제외 재료 입력
- 알레르기 토글 입력

### 7.2 홈
- 현재 식사 조건 입력
- 추천 버튼
- 최근 추천 이력 1개 요약

### 7.3 추천 결과
- 상위 3개 카드
- 카드별 추천 이유 표시
- 좋아요 / 별로예요 피드백 버튼

### 7.4 설정
- 선호도 수정
- 로컬 데이터 초기화

## 8. 품질 요구사항
- 앱 시작 후 홈 진입은 2초 이내
- 추천 계산은 500ms 이내 목표
- 네트워크 없이도 추천 가능
- 제외 재료 및 알레르기 필터는 항상 추천보다 우선 적용

## 9. 제약 조건
| 항목 | 내용 |
| --- | --- |
| 플랫폼 | Android 10 이상 |
| UI | Jetpack Compose |
| 상태 관리 | MVP 범위에서는 MVVM 사용 |
| 추천 엔진 | 로컬 점수 계산 기반, 필요 시 NDK 모듈 사용 |
| 데이터 저장 | Room + DataStore |

## 10. 성공 지표
- 추천 완료율: 추천 버튼 클릭 후 결과 화면 진입 비율 90% 이상
- 재사용성: 사용자가 3일 내 2회 이상 재방문
- 오류율: 추천 계산 중 앱 크래시 0건 목표

## 11. 오픈 이슈
- 추천 데이터셋 초기 규모는 50개 메뉴로 시작할지 100개 메뉴로 시작할지 추후 결정
- 추천 이유 문구를 규칙 기반으로 생성할지 템플릿 기반으로 생성할지 추후 결정
