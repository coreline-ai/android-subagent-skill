# Design Intent - TastePick

## 1. 핵심 목적
`TastePick`의 MVP 목적은 사용자가 짧은 시간 안에 식사 결정을 내릴 수 있도록, 과도한 탐색 없이 상위 3개의 메뉴 후보를 제시하는 것이다.

## 2. 설계 원칙
- 오프라인 우선: 네트워크 없이 추천 가능해야 한다.
- 안전 우선: 알레르기 및 제외 재료 필터는 추천 점수 계산보다 먼저 적용한다.
- 단순한 상태 흐름: MVP는 MVVM 기반으로 구성하고 화면 상태는 `UiState` 단일 구조로 관리한다.
- 교체 가능한 추천 엔진: 추천 점수 계산은 Kotlin 구현으로 시작하되, 동일한 입력/출력을 가진 NDK scorer로 교체 가능해야 한다.

## 3. 주요 유스케이스
### 3.1 온보딩
- 입력: 선호 카테고리, 제외 재료, 알레르기, 매운맛 허용치
- 기대 결과: 이후 추천 요청 시 저장된 사용자 선호도가 자동 반영된다.

### 3.2 추천 요청
- 입력: 식사 시간, 기분, 예산
- 처리:
  1. 사용자 선호도 로드
  2. 메뉴 데이터셋 로드
  3. 제외 재료/알레르기 필터 적용
  4. 조건 기반 점수 계산
  5. 상위 3개 선정
- 기대 결과: 결과 카드에는 추천 이유가 함께 포함된다.

### 3.3 피드백 반영
- 입력: 좋아요/별로예요
- 기대 결과: 동일한 메뉴 또는 카테고리에 대한 후속 추천 점수에 반영된다.

## 4. 아키텍처 의도
- UI와 추천 계산 로직을 분리해, ViewModel이 UI 상태 조합만 담당하게 한다.
- 추천 엔진은 Domain/Data 경계 아래에서 호출되어야 하며, UI 계층에서 JNI 호출을 직접 하지 않는다.
- Room과 DataStore는 로컬 저장소로만 사용하고, 초기 MVP에서는 원격 API 의존성을 두지 않는다.

## 5. 방어 로직 기준
- 추천 결과가 0건이면 빈 리스트가 아니라 "조건 완화 안내 상태"를 반환해야 한다.
- JSON seed 로드 실패 시 앱 전체 크래시 대신 복구 가능한 오류 상태를 보여줘야 한다.
- 알레르기 필터 누락은 Critical 등급의 설계 위반으로 간주한다.

## 6. 상태 관리 기준
- MVP 기준 상태 패턴은 MVVM이다.
- MVI는 본 문서 범위에서 기본 요구사항이 아니다.
- ViewModel은 `StateFlow<UiState>`를 외부로 노출하고 내부 가변 상태는 숨긴다.

## 7. NDK/JNI 적용 기준
- NDK는 추천 점수 계산 최적화가 필요할 때만 사용한다.
- JNI 사용 시 포인터 생명주기, 예외 변환, 백그라운드 스레드 호출 규칙을 준수한다.
- Kotlin 구현과 NDK 구현은 동일한 추천 결과 정렬 기준을 유지해야 한다.

## 8. 리뷰 시 반드시 확인할 항목
- 알레르기/제외 재료가 추천 이전 필터로 구현되었는가
- 추천 결과 3개와 추천 이유가 함께 노출되는가
- MVVM 상태 관리 규칙을 위반하지 않았는가
- 추천 엔진이 UI 계층에 직접 결합되지 않았는가
